# 4-ТАРАУ. ТЕСТІЛЕУ ЖӘНЕ ҚАТЕЛЕРДІ ТҮЗЕТУ

## 4.1 Тестілеудің мақсаты мен әдістемесі

Бағдарламалық қамтамасыз етуді (БҚ) әзірлеу цикліндегі ең маңызды кезеңдердің бірі – оның сапасын бақылау және тестілеу болып табылады. Тәжірибелік зерттеу көрсеткендей, кез келген күрделі веб-қосымшаны, әсіресе электрондық коммерция және жасанды интеллект интеграциясы бар платформаларды енгізу барысында код деңгейінде немесе интерфейстік деңгейде қателердің кездесуі заңдылық. «QarStyle» электрондық коммерция платформасының жұмыс істеу қабілетін, қауіпсіздігін, өнімділігін және пайдаланушыларға ыңғайлылығын (UI/UX) тексеру мақсатында кешенді тестілеу жұмыстары жүргізілді.

Тестілеудің негізгі мақсаттары келесідей анықталды:
1. Жүйенің функционалдық талаптарға толық сәйкестігін тексеру (тіркелу, авторизация, каталогпен жұмыс, себет жүйесі, тапсырысты Kaspi Pay арқылы рәсімдеу, жеке кабинет және әкімшілік панель жұмысы).
2. Пайдаланушы интерфейсінің (UI/UX) әртүрлі экран өлшемдерінде (бейімделгіш дизайн) және әртүрлі шолғыштарда (кроссбраузерлік) дұрыс бейнеленуін бақылау.
3. Платформаның өнімділігін талдау, деректер қорына сұраныстарды оңтайландыру (Django ORM сұраныстарының N+1 мәселесін анықтау және жою).
4. Кіріктірілген ЖИ-ассистенттің (AI Assistant) сұрақтарға жауап беру сапасын, API байланысы үзілгендегі резервтік жүйенің (fallback mechanism) жұмысын тексеру.
5. Тестілеу барысында анықталған қателерді (bugs) жүйелі түрде тіркеу және оларды толықтай жою.

Осы мақсаттарға жету үшін тестілеудің бірнеше әдістері қолданылды:
* **Қара жәшік (Black Box) әдісі**: Платформаның ішкі кодына үңілмей, оның сыртқы интерфейсі мен функционалдық элементтерінің пайдаланушы тұрғысынан дұрыс жұмыс істеуін қолмен тестілеу (Manual Testing).
* **Сұр жәшік (Grey Box) әдісі**: Жүйенің код құрылымын ішінара білу арқылы деректер қорының сұраныстарын (SQL), API жауаптарын және серверлік журналдарды (logs) талдай отырып тестілеу.
* **Автоматтандырылған құралдармен тестілеу**: Браузер өнімділігі мен SEO көрсеткіштерін бағалау үшін Google Lighthouse құралын, ал серверлік сұраныстарды талдау үшін Django Debug Toolbar құралын қолдану.

## 4.3 Веб-интерфейсті (UI/UX) және кроссбраузерлікті тестілеу

Платформаның заманауи және премиум деңгейдегі дизайнын (гармониялық түстер палитрасы, тегіс микро-анимациялар, жылжымалы Sticky-панельдер) сақтау және оның әртүрлі құрылғыларда дұрыс бейнеленуін қамтамасыз ету мақсатында интерфейстік тестілеу жүргізілді.

Тестілеу барысында келесі визуализациялық элементтер тексерілді:
1. **Responsive Web Design**: Платформаның экран ені 1920px (үлкен мониторлар), 1024px (планшеттер) және 375px (мобильді телефондар) өлшемдерінде жұмыс істеуі. CSS Media Queries көмегімен жасалған бейімделудің дұрыстығы тексерілді. Мысалы, мобильді нұсқада бүйірлік сүзгілер жасырын модальді терезеге жинақталып, жоғарғы мәзір «бургер» батырмасына ауысты.
2. **Анимациялар мен интерактивтілік**: Vanilla JavaScript негізінде жазылған өнім сипаттамасы блоктарының тегіс ашылуы (`scrollHeight` және CSS `transition` қолдану арқылы), суреттер слайдерінің жұмысы, батырмалардың үстіне меңзерді апарғандағы (hover effect) реакциялары бақыланды.
3. **Кроссбраузерлік үйлесімділік**: Веб-сайттың әртүрлі браузер қозғалтқыштарында (Blink, WebKit, Gecko) бірдей деңгейде көрсетілуі тексерілді. Тестілеу нәтижесі 4.3.1-кестеде көрсетілген.

## 4.4 Деректер қоры мен өнімділікті тестілеу

Платформаның жүктелу жылдамдығы мен деректер қорымен (SQLite/PostgreSQL) өзара әрекеттесу тиімділігін талдау үшін **Django Debug Toolbar** құралы қолданылды. Алғашқы сынақтар барысында өнімдер каталогы (catalog) бетінде өнімділікке қатысты үлкен мәселе анықталды: тауарлар тізімін және олардың суреттері мен санаттарын жүктеу кезінде **N+1 сұраныс проблемасы** туындады.

Мәселенің мәні: жүйе алдымен барлық тауарларды алу үшін 1 сұраныс жібереді, содан кейін әрбір тауардың санатын, бөлімін және сәйкес суреттерін алу үшін деректер қорына қосымша жеке-жеке сұраныстар орындайды. Каталогта 30 тауар болған жағдайда, деректер қорына 30-дан астам артық сұраныс жіберіліп, бұл сервердің жауап беру уақытын айтарлықтай баяулатты.

Бұл мәселені шешу үшін Django ORM-нің сұраныстарды алдын ала біріктіріп жүктейтін құралдары енгізілді:
* `select_related('category', 'department')` – бір-біріне ForeignKey арқылы байланысқан сыртқы кілттерді бір SQL JOIN сұранысымен біріктіріп жүктеу үшін қолданылды.
* `prefetch_related('images', 'variants')` – кері байланыстарды (бір тауардың бірнеше суреті немесе өлшемдік вариациялары) оңтайландырылған қосымша сұраныс арқылы Python деңгейінде біріктіру үшін енгізілді.

Оңтайландыру жұмыстарының тиімділігін салыстыру көрсеткіштері 4.4.1-кестеде көрсетілген.

### Кесте 4.4.1 – Деректер қоры сұраныстарын оңтайландыру нәтижелері

| Бақыланатын көрсеткіш | Оңтайландыруға дейін (Стандартты ORM) | Оңтайландырудан кейін (Select/Prefetch Related) | Жақсару тиімділігі (%) |
| :--- | :---: | :---: | :---: |
| **SQL сұраныстарының жалпы саны** | 32 сұраныс | 3 сұраныс | **-90.6% (сұраныстар қысқарды)** |
| **Деректер қорын өңдеу уақыты (DB Query Time)** | 120 мс | 12 мс | **-90.0% (жылдамырақ)** |

Сонымен қатар, Google Lighthouse құралы арқылы сайттың клиенттік бөлігіне өнімділік аудиті жүргізілді. Аудит нәтижесінде келесі оңтайландырулар іске асырылды:
1. **Cache Busting**: Браузердегі ескі кэштелген файлдардың жаңаруын қамтамасыз ету үшін статикалық файлдардың URL-іне нұсқа нөмірі қосылды (мысалы, `styles.css?v=17` және `main.js?v=5`). Бұл әдіс клиенттердің стильдер мен скрипттердің тек соңғы нұсқасын жүктеуін қамтамасыз етті.
2. **Суреттерді оңтайландыру**: Барлық жүктелетін тауар суреттері заманауи WebP форматына көшіріліп, өлшемдері сығылды.
3. **Аудит нәтижелері**: Google Lighthouse бойынша көрсеткіштер жоғары деңгейге жетті: Өнімділік (Performance) – **94/100**, Қолжетімділік (Accessibility) – **96/100**, Үздік тәжірибелер (Best Practices) – **98/100**, SEO оңтайландыру – **100/100**.

## 4.5 Жасанды интеллект ассистентін (AI Assistant) тестілеу және сенімділік талдауы

Платформаның басты ерекшеліктерінің бірі – тұтынушыларға тауарларды таңдауға көмектесетін біріктірілген интеллектуалды виртуалды кеңесші болып табылады (`ai_chat` view). Жүйенің сенімді жұмыс істеуін қамтамасыз ету мақсатында ЖИ-ассистентті тестілеу екі бағытта жүргізілді: сұрақтарға жауап беру сапасын бағалау және API үзілістеріне төзімділікті тексеру.

### 1. Сұрақтарға жауап беру сапасы мен жүйе ережелерін тексеру
Жүйеге жіберілетін сұраныстардың (prompts) қауіпсіздігі мен шектеулерін тексеру мақсатында 30 түрлі тесттік сұрақтар жинағы дайындалды. Сынақ барысында келесі ережелердің орындалуы қадағаланды:
* **Тауар ұсыну форматы**: Егер қолданушы сұраған тауар қоймада бар болса, ЖИ оны мәтінмен жазбай, тек `[PRODUCT_ID:X]` форматында тегпен қайтаруы керек. Сонда фронтенд оны оқып, әдемі тауар карточкасын шығарады.
* **Жоқ тауарға жауап беру**: Каталогта жоқ тауарларды сұрағанда (мысалы, аяқ киім немесе бас киімдер), ЖИ тек қана: *"К сожалению, этого товара у нас сейчас нет в наличии."* деп қысқа қайыруы қажет.
* **Тақырыптан тыс (неуместные) сұрақтарды блоктау**: Егер қолданушы саясат, ауа-райы немесе тұрмыстық сұрақтар қойса немесе бейәдеп сөздер жазса, ЖИ оған қатаң түрде: *"Я виртуальный консультант магазина одежды. Пожалуйста, давайте вернемся к выбору гардероба."* деп жауап беруі тиіс.

Тестілеу сценарийлері мен ЖИ-ассистенттің реакциялары 4.5.1-кестеде көрсетілген.

### Кесте 4.5.1 – ЖИ-ассистенттің сұраныстарды өңдеуін тестілеу

| :--- | :--- | :--- | :--- | :---: |

### 2. Көпмодельді фоллбэк жүйесінің (Fallback Chain) жұмысын тестілеу
Жүйенің істен шығуына жол бермеу үшін, `ai_chat` функциясының ішінде сыртқы API сұраныстарына арналған 4 деңгейлі резервтік тізбек (Fallback Chain) әзірленді. Егер негізгі модель уақыт бойынша жауап бермесе немесе шектеулерге тап болса (Rate Limit / Timeout), жүйе келесі модельге автоматты түрде ауысады. Оның орындалу тізбегі келесідей тексерілді:
1. **Google Gemini 1.5 Flash API** (Басты модель – ең жылдам әрі сапалы).
2. **Groq Llama 3.1 8B Instant API** (Бірінші резервтік модель – Gemini API қате бергенде немесе сұраныс уақыты 5 секундтан асқанда іске қосылады).
3. **Groq Mixtral 8x7B API** (Екінші резервтік модель – Llama API істен шыққанда қосылады).
4. **Жергілікті деректер қорынан таңдау (Local DB Fallback)** (Үшінші және соңғы деңгей – егер интернет байланысы толық үзілсе, жүйе қателік көрсетпей, деректер қорынан кездейсоқ қолжетімді тауарды таңдап, дайын шаблонмен ұсынады: *"Здравствуйте! Рекомендую обратить внимание на [Товар] [PRODUCT_ID:X]. Это отличный выбор!"*).

Бұл жүйелік архитектура тестілеу кезінде API желілерін қолдан ажырату арқылы тексеріліп, өзінің 100% тұрақтылығы мен пайдаланушыға ешқандай үзіліссіз қызмет көрсете алатынын дәлелдеді.

## Қорытынды

Жүргізілген кешенді функционалдық, бейімділік, өнімділік және жасанды интеллект интеграциясын тестілеу жұмыстары «QarStyle» электрондық коммерция платформасының толыққанды жұмыс істеуге дайын екенін көрсетті. Өнімділіктегі ең үлкен кедергі болған N+1 сұраныс мәселесі толығымен шешілді, ал ЖИ-ассистенттің көпдеңгейлі фоллбэк архитектурасы API байланыстары үзілгенде де жүйенің тұрақтылығын қамтамасыз ете алатынын растады. Барлық табылған интерфейстік және серверлік қателер толығымен түзетіліп, бағдарламалық өнімнің сапасы мен қауіпсіздігі жоғары деңгейге жеткізілді.
